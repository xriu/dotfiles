# /problem-first

## When to use

- Roadmap is solution-heavy with no clear problem statements
- Stakeholders hand you features to ship, not problems to solve
- Team jumps to implementation before validating user needs
- You need to push back without looking obstructionist

## Quick start

1. Take the **proposed solution** and **context**
2. Ask: *What user job is this trying to fulfill?*
3. Strip the solution. Define the problem in user terms.
4. Verify: does the problem statement stand alone without referencing the solution?

## Workflow

### 1. Isolate the proposal

Write down the solution as given — no editing.

### 2. Backward-map to the job

Ask these in order:

- [ ] What user **outcome** is this meant to enable?
- [ ] What **job-to-be-done** does the user hire this for?
- [ ] What **current workaround** are they replacing?
- [ ] What **trigger** makes them seek this?

### 3. Reframe as problem statement

Template:

> When [situation], [user type] wants to [motivation], so they can [outcome]. Currently, they [friction] because [root cause].

### 4. Pressure-test

- [ ] Can a reasonable person disagree with the problem? (If not, it’s a feature description in disguise.)
- [ ] Does it mention no specific solution?
- [ ] Would solving this actually change user behavior?

### 5. Realign the team

Present: *"We’re aligned on shipping X. Before we scope it, let’s confirm we’re solving the right problem."*

Then share the reframed problem statement and ask for edits.

## Example

**Input:** "Add a dark mode toggle"

**Reframed:** "When using the app at night, professionals want to reduce eye strain so they can work longer without fatigue. Currently, they lower screen brightness or stop using the app after 8pm because the light interface causes discomfort."

Now the team can evaluate whether a toggle, auto-switching, or system-respect is the right solution.
