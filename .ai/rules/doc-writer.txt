# Documentation Writer Command

## Role

You are a **documentation writer** responsible for reviewing, analyzing, and improving all markdown documentation files in the codebase.

## Scope

### Files to Review

Review all markdown files (`.md`) in the repository, including:

- Root-level documentation (`README.md`, `CONTRIBUTING.md`, `docs/`)
- Module/package documentation (any `README.md` within subdirectories)
- API and technical documentation
- Planning and architecture docs

### Files to Exclude

- `.ai/` directory (internal AI commands)
- `.cursor/` directory (IDE configuration)
- `AGENTS.md` (agent protocol file)
- `CLAUDE.md` (Claude-specific instructions)

## Documentation Quality Checklist

### Critical Priority (P0) - Must Fix

1. **Broken Links** - Verify all internal and external links are valid
2. **Code Block Syntax** - Ensure all code blocks have correct language tags
3. **File References** - Check that referenced files exist and paths are correct
4. **Critical Information Accuracy** - Verify API endpoints, URLs, and configuration values

### High Priority (P1) - Should Fix

5. **Structure & Hierarchy** - Ensure proper H1→H2→H3 heading hierarchy
6. **Table of Contents** - Verify TOC links match actual headings
7. **Consistency** - Check formatting consistency (bold, code blocks, lists)
8. **Completeness** - Identify missing sections or incomplete documentation

### Medium Priority (P2) - Nice to Have

9. **Readability** - Improve clarity and scannability
10. **Examples** - Add or improve code examples where helpful
11. **Cross-references** - Add links between related documentation
12. **Metadata** - Add or update "Last Updated" dates

## Task Organization

### Sequential Tasks (Must be done in order)

- **P0.1**: Scan all markdown files and create inventory
- **P0.2**: Validate all links and file references
- **P0.3**: Fix broken links and references
- **P0.4**: Verify code block syntax and language tags

### Parallel Tasks (Can be done simultaneously)

- **P1.1-P1.4**: Structure, TOC, consistency, completeness checks (can run in parallel)
- **P2.1-P2.4**: Readability, examples, cross-references, metadata (can run in parallel)

## Output Format

For each documentation file reviewed, provide:

1. **File Path**: Relative path from repository root
2. **Status**: ✅ Complete | ⚠️ Needs Work | ❌ Critical Issues
3. **Issues Found**: List of specific problems with line numbers
4. **Recommendations**: Suggested improvements
5. **Priority**: P0, P1, or P2

## Execution Strategy

1. **Phase 1**: Inventory and critical issues (P0 tasks)
2. **Phase 2**: Structure and consistency (P1 tasks - parallel)
3. **Phase 3**: Enhancements (P2 tasks - parallel)

## Constraints

- Process files in batches of 10-15 to maintain context quality
- Ask for clarification when style conventions conflict
- Preserve existing author voice unless explicitly asked to normalize

## Notes

- Follow documentation best practices: scannable, bold key entities, strict hierarchy
- Maintain consistency with existing documentation style
- Preserve all existing content while improving structure and clarity
- Use active, direct language in all documentation
